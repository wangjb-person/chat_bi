# ChatBI 问数 Demo（演示版）— 技术选型与实现路线


---

## 实现原理

**定位**：借鉴 **Vanna** 的设计理念，采用**检索增强生成（RAG）**架构实现 **Text-to-SQL** 能力。

**主链路（先检索、再生成、再落地）**

1. **语料入库**：问答+SQL、DDL、文档、通用规则等多类型语料写入向量库并完成嵌入。  
2. **检索增强**：用户问题经同一嵌入模型向量化，在库内按语义召回相关问答先例、表结构、文档与通用知识（可按元数据缩小范围）。  
3. **组装与生成**：将召回内容与系统规则、**示例 Q–SQL**、多轮对话拼成 prompt，由 **LLM 输出 SQL**。  
4. **执行与反馈**：执行 SQL 得到结果，根据结果生成 **图表**、**追问**（仍基于 `question + sql + 结果` 完成）。

**检索在解决什么**：把「与当前问题最相关、体量可控」的上下文交给模型，替代「全量 schema / 全库说明直灌」带来的噪声与超长上下文问题。

## 1. 选取技术框架

| 层次        | 选用技术 | 在项目中的职责                               |
|:----------| :--- |:--------------------------------------|
| Web 与 API | **Flask** | 提供页面与 REST/SSE 接口，单进程演示部署成本低          |
| 数据访问      | **MySQL + PyMySQL + pandas** | 执行模型生成的 SQL，结果以 DataFrame 处理与截断返回     |
| 大模型       | **OpenAI 兼容 SDK（qwen-plus）** | 语言理解与推理，输出一段符合 MySQL 方言、能回答当前问题的查询语句  |
| 向量模型      | **ChromaDB（PersistentClient）** | 持久化存储四类语料向量，按问题语义召回上下文                |
| 文本嵌入      | **sentence-transformers**：`paraphrase-multilingual-MiniLM-L12-v2` | 将问题 / DDL / 文档等编码为向量，支持中英文混合业务表述      |
| 前端        | **原生 HTML/CSS/JS + Plotly.js（CDN）** | 对话、表格、图表渲染，无额外前端工程构建链                 |
| 服务端可视化    | **Plotly（Python）** | LLM 生成作图代码，服务端执行后输出 figure JSON 供前端展示 |
| 配置        | **python-dotenv** | 本地与部署环境通过 `.env` 管理密钥与连接串             |

## 2. 模型与「智能」组件选型及原因

本阶段定位为**本地可跑的简单验证 Demo**：场景与表数量有限、链路不追求复杂能力；算力环境为**普通 PC、无 GPU**，因此嵌入与向量检索优先选 **CPU 友好、开箱即用** 的组合；大模型走 **云端 API** 以避开本地推理卡与显存门槛。**若验证通过再考虑**换更强模型、云向量库或更大 Embedding，当前选型以「能讲清原理、成本低、迭代快」为主。

**（1）大语言模型：通义千问兼容接口（如 qwen-plus）**

- **Demo 与场景：** 该demo问数范围小、不追求极限复杂 SQL，更需要**中文理解 + MySQL 方言生成**的稳定表现。   
- **推理与成本：**  **`qwen-plus` 在通用推理与代码类输出上均衡，响应速度快**；**免费调用额度高，适合反复调试与演示**。
- **工程与后续：** **OpenAI 兼容协议**，便于只改配置即切换其他兼容模型；验证有价值后再评估更强型号或专用 SQL 向模型。

**（2）嵌入模型：paraphrase-multilingual-MiniLM-L12-v2**

- **本地无 GPU：** 模型体量小，**sentence-transformers 在 CPU 上即可做编码**，单次检索延迟在 Demo 语料规模下可接受，无需额外向量推理服务或显卡。  
- **场景与性能：** 语料条数少时，向量维度固定（384）、计算量可控，**吞吐与内存占用**优于大体积句向量模型；多语种设计贴合**中英混排**表名与业务描述。  
- **取舍：** 极端长文档或极专术语场景非本 Demo 重点；若后续扩大语料或对召回要求提高，再考虑更大 Embedding 或云 API 嵌入（需重算向量）。

**（3）向量库：Chroma 本地持久化**

- **Demo 与场景：** 语料量小、无高并发，**单机文件持久化**即可满足「入库—检索—演示」闭环，避免为验证阶段单独部署向量集群。  
- **本地与速度：** 与 Python 同进程集成，**路径即库**，冷启动后检索路径短；按 `sql / ddl / doc / gen` 分集合，与 RAG 问数习惯一致，维护心智成本低。  
- **后续：** 若语料规模、并发或高可用上升，再评估迁移至 Milvus / Qdrant 等，不改变当前主链路原理。

**（4）问数链路中的「非模型」关键能力**

- **SQL 抽取：** 不依赖模型二次调用，用规则从回复中取出可执行语句，**实现轻、延迟低**，适合 Demo 快速稳定出 SQL。  
- **表名过滤：** 场景简单时通过元数据缩小检索范围，**减少无关表噪声**，在有限语料下提升有效上下文比例。

## 3. 市面类型对比与后续升级可考虑方向

### 3.1 实现原理对比（问数主链路）

| 实现原理 | 核心思路 | 相对本项目（RAG + Text-to-SQL） | 适用边界 |
| :--- | :--- | :--- | :--- |
| **RAG + 通用 LLM（当前）** | 先向量检索 DDL / 文档 / 示例 SQL，再让 LLM 生成 SQL | 与 Vanna 理念一致；上下文可控、可解释召回依据 | 表与语料规模中等，需快速 Demo 与持续运营语料 |
| **全量 Schema Prompt** | 不检索或弱检索，将大量表结构直接写入 Prompt | 实现简单，但表多时易超长、噪声大、成本高 | 单库表少、字段稳定的垂直场景 |
| **语义层 / 指标层 + SQL** | 用户问「指标」，先映射到受控指标定义，再生成或拼装 SQL | 口径约束强、幻觉少，需先建设指标与维度模型 | 企业口径统一诉求强、可接受前置建模周期 |
| **领域微调 / LoRA** | 用企业金标 Q–SQL 微调专用模型，弱化对检索的依赖 | SQL 风格与方言更贴业务，依赖数据量与评测体系 | 金标问答对积累到一定规模后，可与 RAG 叠加 |
| **Agent / 多步推理** | 模型多轮：查 schema、试 SQL、纠错、再执行 | 复杂分析链路上限更高，链路长、延迟与成本更大 | 多表关联复杂、需「探索式」分析时评估 |

本项目坚持 **「检索选上下文 → LLM 生成 SQL → 数据库执行」**；上表其余路线多在某一环节替代或增强当前原理，而非否定 RAG 主线。

### 3.2 能力提升时的组件选型方向（LLM / 向量库 / Embedding）

在**不改变 RAG + Text-to-SQL 主原理**的前提下，可按瓶颈分别升级三类组件：

| 组件 | 当前选用 | 升级可考虑方向 | 主要收益与注意点 |
| :--- | :--- | :--- | :--- |
| **LLM** | 通义千问兼容 API（qwen-plus） | 同协议切换：**qwen-max / qwen-coder** 等更强推理或代码向模型；其他国产兼容网关（DeepSeek、智谱等）；开源本地部署（Qwen2.5-Coder、SQLCoder 等经 vLLM/Ollama） | 复杂 JOIN、子查询、方言细节更稳；需回归评测集与成本/延迟评估 |
| **向量数据库** | Chroma 本地持久化 | **Milvus**、**Qdrant**、**Weaviate**；云托管向量服务 | 语料量、并发检索、集群高可用；需迁移脚本与集合元数据设计 |
| **Embedding 模型** | paraphrase-multilingual-MiniLM-L12-v2 | 中文增强：**bge-large-zh**、**m3e-base/large**；多语种/长文本：**bge-m3**；云 API：**DashScope text-embedding**、OpenAI **text-embedding-3-small/large** | 业务术语与中文问句召回更准；换模型后需**重新嵌入全量语料** |

**升级顺序建议（与实现原理相关）：** 语料召回不准时优先换 **Embedding** 或调检索条数；SQL 逻辑错误多时再升 **LLM**；语料规模与 QPS 上来后再换 **向量库** 基础设施。

## 4. 整体问数实现路线

本项目的问数能力按**「检索增强生成（RAG）+ 约束执行」**落地，主链路如下：

1. **语料入库**  
   管理员将 SQL 问答对、DDL、业务说明文档、通用业务规则写入向量库对应集合，并可选绑定 `table_name` / `gen_type` 等元数据，供检索过滤。

2. **在线提问与召回**  
   用户自然语言问题经**同一嵌入模型**编码，在 Chroma 中并行/分别检索：相似问答、相关 DDL、相关文档、通用知识。

3. **提示词组装与 SQL 生成**  
   将系统角色（MySQL 专家）、当日日期、响应规范、召回片段与历史 `messages` 组装为对话，调用 **Chat Completions**（支持流式）；对完整输出做 **SQL 抽取与清洗**，得到可执行语句。

4. **执行与反馈**  
   通过 PyMySQL 在目标数据库执行，结果经 pandas 结构化，返回表格（大行数下截断展示）。

5. **增强能力（可选）**  
   基于当前 `question + sql + df` 调用模型生成 **Plotly 代码** 作图，或生成 **追问列表**。
